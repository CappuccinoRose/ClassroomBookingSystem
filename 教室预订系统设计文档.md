# 教室预订系统业务建模文档

（适配技术栈：Python + Flask + MySQL 8.0 + Vue 3 + Element Plus）

\---

## 一、业务建模

### 1.1 项目概述

#### 1.1.1 项目背景与目标

本系统为面向校园 / 办公场景的教室资源数字化管理平台，核心目标是解决传统线下登记预订效率低、时段冲突频发、使用数据难以统计的痛点，实现教室资源全生命周期线上管理、预订全流程闭环管控、多角色权限隔离。系统采用前后端分离架构，兼顾核心功能的稳定性与扩展功能的灵活性，适配单人开发的工作量与技术能力。

#### 1.1.2 系统功能范围

**核心必选功能**

1. 用户管理：支持用户注册、登录鉴权，内置普通用户、管理员双角色
2. 教室管理：管理员可对教室信息进行新增、编辑、删除与停用维护
3. 教室查询：支持按日期、时间段、容量、设施条件组合筛选空闲教室
4. 预订申请：用户选择教室与时段、填写用途提交预订，系统自动校验时段冲突
5. 我的预订：用户可查看个人所有预订，支持修改预订信息、提前取消预订
6. 管理员后台：全量预订查看、违规预订强制取消、教室使用数据统计

**扩展可选功能**

1. 预订审批流程：普通用户提交的预订需管理员审核通过后方可生效
2. 周期性预订：支持按周固定时段自动生成预订，覆盖长期重复使用场景
3. 冲突检测优化：并发场景下保证同一教室同一时段不可重复预订
4. 邮件通知：预订成功、取消、审核结果等状态变更时自动推送邮件提醒
5. 报表导出：支持导出教室使用率统计报表，兼容 Excel、PDF 格式

### 1.2 业务角色与权限体系

#### 1.2.1 角色定义

* **普通用户**：系统核心使用主体，核心诉求为查询可用教室、提交与管理个人预订，无后台管理权限
* **管理员**：系统运维与资源管理者，负责教室资源维护、预订全量管控、数据统计与合规审核

#### 1.2.2 权限矩阵

|功能模块|操作项|普通用户|管理员|
|-|-|-|-|
|用户账号|注册、登录、修改个人信息|✅|✅|
|教室信息|查看教室列表与详情|✅|✅|
|教室信息|新增、编辑、删除 / 停用教室|❌|✅|
|教室查询|多条件筛选空闲教室|✅|✅|
|预订操作|提交个人预订申请|✅|✅|
|预订操作|查看、修改、取消个人预订|✅|✅|
|预订管控|查看全平台所有预订记录|❌|✅|
|预订管控|强制取消任意预订|❌|✅|

|数据统计|导出使用率报表文件|❌|✅|

### 1.3 核心业务流程

#### 1.3.1 用户注册与认证流程

1. 用户提交账号、密码、邮箱信息完成注册，系统校验账号与邮箱唯一性，默认赋予普通用户角色
2. 用户凭账号密码登录，系统校验身份后发放访问凭证
3. 系统根据用户角色展示对应功能菜单与数据范围，越权操作直接拦截

#### 1.3.2 教室资源管理流程（管理员）

1. 管理员进入教室管理页面，可查看全量教室列表，支持按编号、楼层、状态筛选
2. 新增教室：填写教室编号、楼层、容纳人数、设施配置等信息，提交后系统校验编号唯一性并生效
3. 编辑教室：修改教室基础信息与设施状态，提交后实时同步
4. 删除 / 停用教室：校验当前教室无进行中的有效预订后，可物理删除或标记为停用状态

#### 1.3.3 空闲教室查询与预订主流程

1. 用户选择预订日期、起止时段、最小容量、所需设施等筛选条件，提交查询
2. 系统先筛选出符合基础属性（容量、设施、状态）的教室集合
3. 系统匹配当日已有有效预订，排除存在时段冲突的教室，最终返回空闲教室列表
4. 用户选定目标教室，填写预订用途，提交预订申请
5. 系统二次校验目标教室该时段的可用性，无冲突则生成有效预订记录；存在冲突则驳回申请并提示原因

#### 1.3.4 预订修改与取消流程

1. 用户进入「我的预订」列表，选择目标预订记录
2. **取消预订**：系统校验当前时间与预订开始时间的间隔，间隔≥40 分钟则允许取消，更新预订状态并释放教室资源；间隔不足 40 分钟则拦截操作，提示需联系管理员处理
3. **修改预订**：用户修改时段、用途等信息后提交，系统按新预订规则重新执行冲突校验；校验通过则更新原预订记录，校验失败则保留原预订不变并提示原因

#### 1.3.5 预订审批流程（扩展）

1. 开启审批模式后，普通用户提交的预订默认状态为「待审核」，暂不占用教室资源
2. 管理员在后台待审核列表中查看预订详情，核对用途与使用合规性
3. 管理员执行通过操作：预订状态更新为「已确认」，正式锁定教室时段
4. 管理员执行驳回操作：预订状态更新为「已驳回」，释放教室占位，需填写驳回原因
5. 审批完成后，系统同步通知用户审批结果

#### 1.3.6 周期性预订流程（扩展）

1. 用户创建周期预订规则，指定目标教室、每周固定星期几、固定时段、用途、规则生效起止日期
2. 系统校验规则基础合法性，以及规则有效期内首个时段的可用性
3. 系统按规则定期批量生成未来对应日期的预订记录，逐条执行冲突检测
4. 无冲突的预订直接生效；存在冲突的日期跳过生成并记录异常
5. 用户可停用周期规则，后续不再生成新预订，已生成的有效预订不受影响

#### 1.3.7 统计与报表导出流程（扩展）

1. 管理员选择统计时间范围与统计维度（按教室、按楼层、按时段）
2. 系统聚合有效预订数据，计算各教室的预订总时长、使用率、热门时段分布
3. 系统以图表形式展示统计结果，支持导出为 Excel 明细报表或 PDF 汇总报表

### 1.4 业务规则约束

#### 1.4.1 时段冲突检测规则

* 核心判定逻辑：两个预订时段存在重叠，即判定为冲突。重叠判定标准：新预订开始时间早于已有预订结束时间，且新预订结束时间晚于已有预订开始时间
* 有效检测范围：仅状态为「待审核」「已确认」的预订参与冲突校验，已取消、已驳回、已完成的预订不占用时段资源
* 并发约束：同一教室同一时段仅允许存在一条有效预订，禁止重复预订

#### 1.4.2 预订取消时限规则

* 用户自主取消：需在预订开始时间前 40 分钟以上操作，超时后无法自主取消
* 管理员强制取消：无时间限制，可随时取消任意预订，操作需留痕
* 取消生效后，教室对应时段即时释放，可被其他用户重新预订

#### 1.4.3 审批业务规则

* 仅普通用户提交的预订需审核，管理员提交的预订直接生效
* 审批需在预订开始时间前完成，超时未审核的预订可配置自动驳回
* 审批通过时需二次校验时段可用性，避免审核期间时段被其他预订占用

#### 1.4.4 周期预订生成规则

* 周期粒度为按周重复，支持选择每周固定某一天
* 仅生成未来日期的预订，不回溯生成历史记录
* 单条日期预订冲突不影响其他日期的生成，异常情况单独记录
* 规则停用后终止新预订生成，不影响已生成的有效预订

#### 1.4.5 使用率统计规则

* 单教室使用率 = 统计周期内已确认预订总时长 / 统计周期内可预订总时长 × 100%
* 统计维度支持按日、周、月聚合，支持按楼层、按容量分组
* 热门时段按小时粒度统计预订次数，输出时段热度分布

## 二、数据建模

### 2.1 领域实体模型

#### 2.1.1 核心实体定义

|实体名称|业务含义|核心属性|
|-|-|-|
|用户|系统操作主体，包含普通用户与管理员两类角色|用户 ID、账号、密码、邮箱、角色、账号状态|
|教室|可预订的物理资源主体|教室 ID、编号、楼层、容量、设施配置、状态|
|预订记录|单次教室预订的业务凭证|预订 ID、用户 ID、教室 ID、日期、起止时间、用途、状态|
|审批记录|预订审核的操作留痕（扩展实体）|审批 ID、预订 ID、管理员 ID、审批结果、意见、时间|
|周期规则|周期性预订的生成规则（扩展实体）|规则 ID、用户 ID、教室 ID、周期星期、时段、有效期、状态|

#### 2.1.2 实体关系说明

1. 用户与预订记录：一对多（1:N），一个用户可创建多条预订记录
2. 教室与预订记录：一对多（1:N），一个教室可对应多条历史预订记录
3. 预订记录与审批记录：一对一（1:1），一条预订对应唯一一条审批记录
4. 用户与周期规则：一对多（1:N），一个用户可创建多条周期规则
5. 教室与周期规则：一对多（1:N），一个教室可对应多条周期规则
6. 周期规则与预订记录：一对多（1:N），一条规则可生成多条预订记录

#### 2.1.3 ER 关系总览

```Plain Text
用户 1────N 预订记录 N────1 教室
  │            │
  │            1
  │            │
  N            1
审批记录    周期规则 1────N 预订记录
```

### 2.2 数据库设计规范

* **存储引擎**：统一使用 InnoDB，支持事务、行级锁与外键约束，保障预订操作的原子性与数据一致性
* **字符集**：全局采用`utf8mb4`，排序规则为`utf8mb4\_unicode\_ci`，兼容全量中文与特殊字符
* **命名规范**：表名、字段名统一使用小写蛇形命名；主键统一命名为`xxx\_id`；状态类字段统一用 TINYINT 编码
* **事务约定**：预订提交、取消、审批等核心写操作必须包裹在事务中，保证数据原子性

### 2.5 索引设计方案

|表名|索引名称|索引字段|索引作用|
|-|-|-|-|
|reservation|idx\_room\_date\_status|classroom\_id, reserve\_date, status|核心冲突检测与空闲查询，大幅提升检索速度|
|reservation|idx\_user\_create|user\_id, create\_time DESC|个人预订列表分页查询优化|

|periodic\_rule|idx\_weekday\_status|status, weekday, classroom\_id|定时任务批量检索规则，优化生成效率|

### 2.6 数据完整性约束

1. **外键约束**：所有关联字段设置外键，保证用户、教室、预订、规则之间的关联一致性，避免无效脏数据
2. **CHECK 约束**：预订表设置`end\_time > start\_time`，周期规则表设置`end\_date > start\_date`，保证业务逻辑合法
3. **唯一约束**：用户账号、邮箱、教室编号全局唯一，避免重复数据
4. **事务保障**：预订提交、取消、审批等核心操作使用数据库事务，出现异常自动回滚，保证数据原子性

## 三、接口规范

### 3.1 接口总体规范

#### 3.1.1 设计风格

采用 RESTful API 设计风格，以资源为核心，使用标准 HTTP 方法表达操作语义：GET（查询）、POST（新增）、PUT（修改）、DELETE（删除）。接口统一前缀为`/api`，版本号通过路径区分，如`/api/v1`。

#### 3.1.2 统一响应格式

所有接口返回统一 JSON 结构，便于前端统一处理：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

* `code`：业务状态码，200 表示成功，其他表示异常
* `message`：操作结果描述
* `data`：响应业务数据，无数据时为 null

#### 3.1.3 鉴权方式

采用 JWT（JSON Web Token）鉴权机制：

* 用户登录成功后获取 access\_token，后续请求在 Header 中携带`Authorization: Bearer <token>`
* 后端统一拦截校验 Token 有效性与用户角色，越权请求直接返回 403
* Token 设置有效期，超时后需重新登录

#### 3.1.4 通用状态码约定

|状态码|含义|
|-|-|
|200|操作成功|
|400|参数错误|
|401|未登录 / Token 无效|
|403|无权限访问|
|404|资源不存在|
|409|业务冲突（如预订时段冲突）|
|500|服务器内部错误|

### 3.2 用户认证模块接口

|接口路径|请求方式|接口说明|权限|
|-|-|-|-|
|/api/auth/register|POST|用户注册|匿名|

|/api/auth/userinfo|GET|获取当前登录用户信息|登录用户|
